# Phase 1 Implementation Log

## Overview
In Phase 1, we established the core foundation for the CaptionLayer Desktop Application. The primary goal was to initialize the project, establish the canonical data structures, and build a foundational UI shell.

### What We Did
1. **Architecture & Planning:** Documented our plan in `docs/10-implementation-plan.md` focusing on a local-first logic, deterministic + Gemini segmentation, and using Pydantic, PySide6, and WhisperX.
2. **Project Initialization:** Bootstrapped the Python project using `uv init` inside the `desktop-app` directory and added core dependencies (`pyside6`, `whisperx`, `pydantic`).
3. **Canonical Data Model:** Modeled our universal timeline structure (`Word`, `Timing`, `WordStyle`, `Segment`, `SegmentStyle`, `CanonicalTimeline`) using Pydantic in `src/domain/schemas/timeline.py`.
4. **UI Shell Generation:** Created the core dark-theme creator interface in PySide6 with a project sidebar and placeholder timeline list viewer (`src/ui/windows/main_window.py`).
5. **Entry Point Integration:** Replaced the autogenerated `main.py` setup to properly append the `src/` path to `sys.path` and launch the PySide6 app.

## Errors & Bumps Encountered
1. **Working Directory Mismatch:** Attempted to run `uv add` from the root workspace directory before changing into the `desktop-app` folder. This naturally threw an error (`error: No pyproject.toml found`). 
2. **PowerShell `mkdir -p` Incompatibilities:** Attempted to use Bash-like syntax `mkdir -p a b c` inside the Windows PowerShell environment. PowerShell interpreted it incorrectly, throwing a ParameterBindingException. 
3. **Overwriting Existing Files:** Tried to use a file creation tool to write `main.py` without realizing that `uv init` automatically creates a placeholder `main.py`. This resulted in a "File already exists" error.
4. **Race Condition with Dependency Install:** Attempted to execute the application (`uv run ...`) while PySide6 and Torch were still installing in a background terminal, leading to a `ModuleNotFoundError: No module named 'PySide6'`. 

## How We Overcame Them
- Fixed package management by properly targeting `cd "desktop-app"` before running `uv` dependency updates in a background process (since Torch/WhisperX takes time).
- Swapped the `mkdir` command for proper Windows PowerShell syntax: `New-Item -ItemType Directory -Force -Path "dir1", "dir2"`.
- Utilized string replacement logic instead of wholesale file creation to update the existing `main.py`.
- Recognized that background installations need to fully resolve before runtime testing.

## Silly Mistakes & Learnings for Phase 2
- **Check Before Creating:** We need to inspect what template files (`uv init` generated files) exist before trying to create them blindly.
- **PowerShell Awareness:** We are running on a Windows environment using PowerShell 5.1. Always prefer native/cross-platform Python scripts or PowerShell idiomatic commands (`New-Item`, `Get-ChildItem`) over Bash-isms.
- **Be Patient with Installs:** AI modules (like torch via WhisperX) are massive. When we work on Phase 2 (triggering WhisperX), we have to ensure the environment is fully built and remember that WhisperX model downloads will also take time on first execution.
- **Context Boundaries:** The workspace root is `lucid-kepler`, but our python universe is contained entirely within `lucid-kepler/desktop-app`. We must maintain strict path awareness when invoking commands.
